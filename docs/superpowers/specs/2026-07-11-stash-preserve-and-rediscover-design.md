# Stash — Preserve and Rediscover Design

**Date:** 2026-07-11  
**Status:** Approved direction; awaiting written-spec review

## Product promise

Stash preserves the useful essence of every saved TikTok so it remains understandable, searchable, and reusable even if the original video is deleted. The original TikTok is provenance and an optional way back, not the permanent source of truth.

The next release adds five features in this order:

1. Offline essence cards
2. Adaptive visual keepsakes
3. Semantic search
4. Automatic cross-topic connections
5. A personalized rediscovery home feed

The design is preservation-first: preserve well, then use the preserved material for retrieval, connections, and rediscovery.

## 1. Offline essence cards

Every successfully imported video becomes a durable **Stash Card**. Its visual hierarchy is essence-first: the cover or motion preview supports the extracted idea rather than dominating it.

A card contains:

- a one-sentence essence;
- category-specific structured information;
- two to four locally stored key images when useful;
- one locally stored three-to-six-second motion excerpt only when motion is instructional;
- a category-appropriate cover;
- the searchable transcript, OCR text, topics, and semantic embedding;
- creator attribution and the original TikTok URL;
- preservation state for text, structure, images, and motion.

Existing category payloads remain supported. Recipe extraction expands to include optional portions, preparation time, and nutrition facts when the source provides enough evidence. Values must not be invented. Music cards use resolved album artwork and do not preserve a motion excerpt by default.

The detail screen presents the preserved card first. The original link appears as a secondary action. If the original becomes unavailable, the card remains fully functional and displays a discreet source-unavailable state.

## 2. Adaptive visual keepsakes

The media stage keeps its temporary audio and frames until analysis identifies the important moments. A new preservation stage then chooses durable assets according to the analyzed content:

- **Music:** album artwork, track, artist, universal link, and a short contextual note when one is supported by the source.
- **Recipe or meal prep:** finished-dish cover, key technique frames, and a motion excerpt only for an instructional movement.
- **Tutorial or demonstration:** before/after and critical intermediate frames, with an optional motion excerpt.
- **Product, place, or inspiration:** identifying visuals, extracted names or links, and the central recommendation.
- **General knowledge:** representative frames, distilled claim, supporting details, and topics.

Transcript and OCR evidence include timestamps. The analyzer returns important time ranges and whether each range requires motion. AVFoundation selects frames near those ranges and exports compressed still images plus, when requested, a short muted loop. The first implementation may use a short looping MP4 rather than GIF because iOS encodes and plays it more efficiently; the user experience is the same.

Durable assets live inside app-owned Application Support storage under a stable directory derived from the video ID. SwiftData stores relative asset paths, never temporary absolute URLs. A manifest records asset type, source timestamp, dimensions, byte size, and creation date. Temporary video, audio, and frames are deleted only after the durable assets and manifest have been written successfully.

Reprocessing is non-destructive: a new asset set is written separately and atomically replaces the old manifest only after success. Failed or partial extraction retains the last valid card.

## 3. Semantic search

Search covers title, essence, structured payloads, caption, transcript, OCR, topics, creator, and extracted entities. It combines:

1. semantic similarity to the query;
2. matches in the essence and structured fields;
3. exact lexical matches for names, ingredients, artists, creators, and OCR text;
4. recency as a light tie-breaker.

Embeddings are generated by the configured model box and stored locally on each card. Query embedding uses the same model. Search remains usable when the box is offline: exact and token-based local search still works, while semantic matching uses previously stored card embeddings and requires only a query embedding. If no query embedding can be generated, the UI labels the result set as text-only rather than failing.

Search is available from the home screen and as its own tab. Results show the matching card essence, cover, category, and a concise reason such as a matched ingredient or related meaning.

## 4. Automatic cross-topic connections

Connections are typed and explainable. Supported initial relation types are:

- same entity, such as song, product, place, ingredient, or technology;
- same technique;
- same goal or problem;
- complementary idea;
- alternative approach.

Candidate pairs come from nearest semantic neighbors. The analyzer validates only those candidates and returns a relation type, short explanation, and confidence. Stash stores a connection only above a conservative threshold. Generic similarity is not sufficient.

Connections are recalculated for a new or reprocessed card against existing cards. Existing unrelated pairs do not require a full-library rebuild. Each card shows a small **Related saves** section with the explanation. Users can dismiss an incorrect connection; dismissed pairs remain suppressed across recalculation.

## 5. Personalized rediscovery home feed

The app opens on a calm **Today** screen titled “Worth returning to.” It contains up to three items per day and has no endless scrolling, autoplay, streaks, urgency, or engagement pressure. Libraries with fewer than three eligible cards show every eligible card without filler.

The daily set balances:

- older cards that have not been revisited;
- cards related to recent saves or searches;
- high-quality cards with complete preserved assets;
- category diversity;
- exclusions for cards shown too recently.

Every item includes an explanation such as “Saved three months ago” or “Related to your recent meal-prep save.” A connected-ideas row may accompany the featured card. The set is generated deterministically from local data for the current calendar day and remains stable until the next day. No notification system is included.

The tab order becomes Today, Library, Search, and Mind Map. Import moves from a dedicated tab to a prominent toolbar action in Library; import progress appears as a persistent Library banner while work is active.

## Architecture and data flow

The existing persisted pipeline remains resumable and best-effort:

```text
Import → Enrich → Fetch media → Transcribe + timestamp → OCR + timestamp
       → Analyze essence/structure/moments/entities
       → Preserve selected assets atomically
       → Embed card
       → Build connections
       → Store complete or partial card
```

The new units have narrow responsibilities:

- `AssetPreserver` selects and writes durable images and motion excerpts.
- `AssetStore` owns stable paths, manifests, atomic replacement, and deletion.
- `EmbeddingClient` calls the configured model-box embedding endpoint.
- `SearchIndex` performs local lexical and vector ranking.
- `ConnectionBuilder` proposes, validates, and stores typed relations.
- `RediscoverySelector` deterministically chooses the daily three cards.

All model-box clients remain protocol-backed so fixtures and fakes can cover the pipeline without network access.

## Data model changes

`Video` gains:

- `essence: String`
- encoded preservation state and asset manifest reference;
- encoded embedding vector plus embedding model/version;
- `lastViewedAt`, `viewCount`, and `lastRediscoveredAt`;
- optional extracted entity records used for exact connections.

A new `CardConnection` SwiftData model stores source video ID, target video ID, relation type, explanation, confidence, creation date, and dismissal state. Connections use a canonical ordered pair so duplicate reverse edges cannot be created.

Asset binaries remain on disk rather than inside SwiftData. Deleting a card deletes its asset directory and connections in one user-visible operation, with orphan cleanup on the next launch if disk deletion fails.

## Error handling

- Original unavailable: retain the card and all local assets; disable only the original-link action.
- Model box unavailable: preserve already selected frames when possible, mark embedding or connection work as awaiting the box, and resume later.
- Asset export failure: keep the prior asset set; never replace it with a partial set.
- Insufficient evidence: omit uncertain structure, entities, or motion rather than inventing content.
- Storage pressure: show estimated Stash storage in Settings and allow deletion by card; automatic destructive cleanup is out of scope.
- Embedding model change: record the model version and lazily re-embed cards in the background.
- Corrupt or missing local asset: render remaining card content, mark the asset missing, and offer reprocessing if the source is still reachable.

## Testing

- Unit tests for timestamp-based frame selection, motion-required decisions, stable asset paths, manifests, atomic replacement, and orphan cleanup.
- Golden tests for category-specific essence and preservation plans using canned analyzer output.
- Search tests for exact, semantic, blended ranking, and text-only fallback.
- Connection tests for candidate limits, thresholds, canonical pairs, explanations, and dismissed-pair suppression.
- Rediscovery tests for a stable daily result, three-item limit, diversity, cooldown, and small-library behavior.
- Migration test from the current `Video` schema with existing saved data.
- Integration test from a fixture video through analysis, durable preservation, embedding, connection building, and SwiftData persistence.
- UI tests for the essence-first card, unavailable original, search results, related saves, and Today feed.

## Acceptance criteria

The release is accepted when:

1. A processed card remains understandable and visually useful with network access disabled and the source URL unavailable.
2. Recipe, music, tutorial, and general saves produce the appropriate balanced visual form without unsupported invented details.
3. Meaning-based queries retrieve relevant cards that do not contain the exact query wording.
4. Displayed connections include a specific human-readable reason and can be dismissed.
5. Today shows a stable, finite set of at most three useful cards and explains why each appeared.
6. Existing imports and current category views survive schema migration.

## Out of scope

Full original-video archiving, cloud sync or backup, sharing, collaborative libraries, notifications, chat-based Q&A, automatic task creation, and unlimited feed generation are excluded from this release.
